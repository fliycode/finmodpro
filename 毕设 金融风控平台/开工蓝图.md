好，我把它收成一版**可以直接开干的毕业设计平台方案**。

我建议你先锁定这条主线：**单仓库、前后端分离、后端分层，不做重微服务**。前端用 Vue；后端用 Django + Django REST Framework 做 API；异步任务用 Celery + Redis；大模型编排用 LangChain，但第一期只做 **2-step RAG + 结构化抽取**，不要一开始上 agent；本地模型先通过 Ollama 接入，因为它提供 OpenAI 兼容接口，后面你切本地模型时最省改动。Milvus Lite 适合你先本地快速起库，后续也能迁到完整 Milvus 服务端，客户端代码变化不大。([Django REST Framework](https://www.django-rest-framework.org/tutorial/quickstart/?utm_source=chatgpt.com))

## 1. 先定总架构

```text
Vue 前端
  ├─ 登录/权限
  ├─ 知识库管理
  ├─ 智能问答
  ├─ 风险抽取
  ├─ 风险报告
  └─ 模型与评测

Django API 层
  ├─ auth 权限
  ├─ kb 知识库
  ├─ chat 问答
  ├─ risk 风险抽取
  ├─ report 报告生成
  ├─ model 模型配置
  └─ eval 评测记录

异步任务层
  ├─ 文档解析
  ├─ 文本切块
  ├─ 向量化
  ├─ 索引重建
  ├─ 风险抽取
  └─ 报告生成

数据层
  ├─ MySQL：业务数据
  ├─ Redis：缓存/队列/任务状态
  ├─ 文件存储：PDF/年报/研报/新闻原文
  ├─ Milvus：向量索引
  └─ Ollama：本地大模型服务
```

这里最关键的一刀是：**MySQL 只管业务数据，不管向量检索**。
也就是说，用户、知识库、文档、会话、抽取结果、评测记录都进 MySQL；向量和检索只进 Milvus。这样你后面替换检索方案时，不会动主业务库结构。Milvus 官方文档现在既支持本地轻量启动，也支持后续迁移；它还支持多向量/混合检索思路，适合你后面把“语义召回 + 关键词召回”做进去。([Milvus](https://milvus.io/docs/quickstart.md?utm_source=chatgpt.com))

## 2. 平台功能模块怎么拆

### 模块 A：用户与权限

第一版只做两类角色就够了：

- 管理员：admin，管理知识库、模型、任务
- 分析师：analyst，问答、抽取、查看报告

别一开始做复杂 RBAC，毕业设计没必要。先做到“页面级 + 接口级权限”就行。

### 模块 B：知识库管理

这是平台的底座，建议做成以下流程：

1. 新建知识库
2. 上传文档
3. 文档解析
4. 文本切块
5. 生成向量
6. 建立索引
7. 查看状态与失败原因

知识来源可以先支持三类：

- 金融年报 PDF
- 研报 PDF / DOCX / TXT
- 新闻文本或网页抓取后的文本

第一期不追求“万能解析器”，只要把 PDF 跑通就已经足够支撑答辩。

### 模块 C：智能问答

问答页别只返回一段答案，要返回 4 块内容：

1. **回答内容**
2. **引用来源**
3. **命中的原文片段**
4. **回答置信提示**

也就是页面不要做成普通聊天框，要做成“**答案 + 证据链**”模式。
因为你的课题核心就是解决金融领域幻觉问题，所以“可追溯”比“看起来像 ChatGPT”更重要。

### 模块 D：风险信息抽取

这个模块不要设计成“让模型自由写一段风险分析”，而要做成**结构化抽取**。

建议抽取结果统一成这种字段：

- company_name：公司名
- report_period：报告期
- risk_type：风险类别
- risk_level：高/中/低
- event_time：事件时间
- summary：风险摘要
- evidence_text：证据原文
- source_doc_id：来源文档
- source_chunk_id：来源片段
- confidence：模型置信
- review_status：待审核/已确认/已驳回

LangChain 现在支持把模型输出约束成 JSON、Pydantic 等结构化结果；这个能力非常适合你做“风险事件表”落库，而不是靠正则从自然语言里硬抠字段。([LangChain 文档](https://docs.langchain.com/oss/python/langchain/structured-output?utm_source=chatgpt.com))

### 模块 E：风险报告与总结

这个模块基于已经抽取出的 `risk_event` 做二次聚合，不要直接拿整库文本去写长报告。

建议两种入口：

- 按公司生成风险摘要
- 按时间区间生成风险摘要

这样你的报告中心可以展示：

- 公司基本信息
- 当前命中的风险事件列表
- 高频风险类别统计
- 模型生成总结
- 引用来源

### 模块 F：模型与评测

这个模块是你后期把“平台”与“模型训练”接起来的桥。

至少要有：

- 当前模型列表
- 模型切换
- Embedding 模型配置
- 提示词模板配置
- RAG 参数配置
- 评测结果展示

这个页面在答辩时很有用，因为老师会问你：
“你换一个模型效果如何？”
“微调前后有什么差异？”
“知识库召回对结果影响多大？”

## 3. 后端工程怎么分 app

我建议 Django 按 app 拆，不按表随便堆：

- `accounts`：登录、用户、角色
- `kb`：知识库、文档、切块、任务
- `chat`：会话、消息、问答接口
- `risk`：抽取、审核、风险事件
- `report`：报告生成与导出
- `llm`：模型适配、Prompt、LangChain 封装
- `eval`：评测集、评测结果
- `system`：系统配置、日志、健康检查

重点是：
**`llm` app 不直接处理页面逻辑，只负责能力封装。**
比如：

- `LLMProvider`
- `EmbeddingProvider`
- `RetrieverService`
- `RiskExtractor`
- `ReportGenerator`

这样以后你从“云模型”切到“本地模型”，或者从“普通问答”切到“LoRA 微调模型”，前端和业务接口不用重写。

## 5. RAG 链路具体怎么做

你的问答链路建议定成：

```text
用户提问
 -> 问题预处理
 -> 关键词召回 + 向量召回
 -> 结果合并/重排
 -> 选 Top-K 片段
 -> 大模型生成答案
 -> 返回答案 + 引用来源 + 命中片段
```

第一版不需要做复杂 Agent。
LangChain 官方把 2-step RAG 定义为“检索固定发生在生成前”，优点就是控制强、延迟可预测；这对毕业设计平台尤其重要，因为你要的是稳定可演示，不是炫技。([LangChain 文档](https://docs.langchain.com/oss/python/langchain/retrieval?utm_source=chatgpt.com))

你的抽取链路则建议定成另一条：

```text
选择文档
 -> 切块遍历
 -> 调用结构化抽取链
 -> 输出 risk_event JSON
 -> 落库
 -> 人工审核
 -> 聚合生成报告
```

也就是说：
**问答链和抽取链分开。**
不要试图用同一条“万能 Agent”同时解决问答、抽取、总结，这样后面特别难调。

## 9. 你这个题目最容易踩的 3 个坑

### 坑 1：平台没问题，但 AI 结果不可追溯

解决办法：
所有问答都必须返回来源片段；所有抽取都必须保存 `evidence_text`。

### 坑 2：抽取结果看起来很多，但没法审核

解决办法：
给 `risk_event` 加 `review_status`，把系统定位成“辅助分析师”，不是替代分析师。

### 坑 3：一开始就想自己训练模型

解决办法：
先把平台链路跑通，再做 LoRA。
答辩老师通常更认可“平台完成度 + 可验证实验”，而不是只讲训练但系统没跑起来。
