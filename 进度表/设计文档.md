# 面向金融风控领域的领域大模型应用平台设计文档

- 日期：2026-03-28
- 课题：面向金融风控领域的领域大模型应用平台构建
- 技术栈：Vue + Django + MySQL + Redis + Celery + LangChain + Milvus + Ollama
- 设计阶段：平台优先，模型训练后置

## 1. 项目背景与问题定义

针对通用大模型在专业金融领域知识不足、回答易出现“幻觉”、风控信息提取不稳定的问题，本项目拟构建一个面向金融风控场景的领域大模型应用平台。平台通过知识库管理、RAG 检索增强问答、结构化风险信息抽取、风险摘要报告生成等能力，为金融分析师提供可追溯、可审核、可扩展的辅助决策工具。

本项目采用“两阶段路线”：

1. **第一阶段：平台能力建设**
   完成用户登录、知识库管理、文档解析、向量索引、问答、风险抽取、风险报告等业务闭环。
2. **第二阶段：模型能力增强**
   使用 LoRA 等参数高效微调方法，基于金融年报、研报、新闻语料完成领域适配，并将本地模型接入平台，对比微调前后的效果差异。

## 2. 建设目标

### 2.1 总体目标
构建一个支持金融文档入库、知识检索、智能问答、风险事件抽取与总结的领域大模型应用平台，并验证其在金融风控垂直场景中的落地可行性。

### 2.2 具体目标
1. 支持金融年报、研报、新闻等文档的上传、解析、切分、索引管理。
2. 支持基于知识库的问答，输出答案与证据来源，降低幻觉风险。
3. 支持对金融文本中的风险信息进行结构化抽取与落库。
4. 支持基于风险事件聚合生成公司级或时间区间级风险报告。
5. 支持后续接入本地模型、微调模型和效果评测模块。

## 3. 系统范围与版本边界

### 3.1 第一版（MVP）范围
第一版只实现以下核心能力：

- 用户登录与权限控制
- 知识库创建与文档上传
- 文档解析、切块、向量化、索引入库
- 基于知识库的智能问答
- 风险信息结构化抽取
- 风险摘要报告生成
- 模型配置与基础评测展示

### 3.2 暂不纳入第一版的能力
以下内容不进入第一版，以避免项目过度膨胀：

- 多租户复杂权限系统
- 实时网页爬虫全链路
- 复杂 Agent 编排
- 多模型自动路由
- 高并发分布式微服务拆分
- 大规模在线训练平台

### 3.3 建议答辩主线
本课题建议采用如下展示主线：

- **主线**：平台工程落地
- **亮点**：金融风控业务能力
- **补充实验**：LoRA 微调前后效果对比

## 4. 总体架构设计

### 4.1 总体架构

```text
Vue 前端
  ├─ 登录与权限
  ├─ 知识库管理
  ├─ 智能问答
  ├─ 风险抽取
  ├─ 风险报告
  └─ 模型与评测

Django API 层
  ├─ accounts 用户认证
  ├─ kb 知识库管理
  ├─ chat 问答服务
  ├─ risk 风险抽取
  ├─ report 报告生成
  ├─ llm 模型适配
  ├─ eval 评测模块
  └─ system 系统配置

异步任务层
  ├─ 文档解析
  ├─ 文本切块
  ├─ 向量化
  ├─ 索引构建
  ├─ 风险抽取
  └─ 报告生成

数据与服务层
  ├─ MySQL：业务数据
  ├─ Redis：缓存/队列/任务状态
  ├─ 文件存储：原始文档
  ├─ Milvus：向量索引
  └─ Ollama：本地模型服务
```

### 4.2 架构原则

1. **前后端分离**：前端负责交互，后端负责业务与 AI 编排。
2. **后端分层**：Django 负责业务 API；模型调用与检索能力做服务封装。
3. **平台优先**：先打通完整业务链路，再接入微调模型。
4. **能力解耦**：问答链、抽取链、报告链分开设计，避免万能链路难以维护。
5. **可追溯**：所有问答和抽取结果必须保留来源证据。
6. **可替换**：模型后端、Embedding 模型、向量库适配尽量通过抽象层隔离。

## 5. 功能模块设计

### 5.1 用户与权限模块

#### 功能
- 用户登录/退出
- 获取当前用户信息
- 角色区分（管理员、分析师）

#### 角色设计
- `admin`：可管理知识库、模型、任务、评测
- `analyst`：可进行问答、抽取、查看报告

#### 第一版限制
不设计复杂 RBAC，只做页面级和接口级权限控制。

### 5.2 知识库管理模块

#### 核心流程
1. 新建知识库
2. 上传文档
3. 文档解析
4. 文本切块
5. 向量化
6. 写入索引
7. 查看状态与失败原因

#### 支持文档类型
- 年报 PDF
- 研报 PDF / DOCX / TXT
- 新闻文本

#### 页面能力
- 知识库列表
- 知识库详情
- 文档列表
- 文档状态页
- 重建索引操作

### 5.3 智能问答模块

#### 目标
基于知识库进行检索增强问答，尽量减少金融领域幻觉。

#### 返回内容要求
问答接口必须返回四类结果：

1. 回答内容
2. 引用来源
3. 命中的原文片段
4. 基础置信提示或召回信息

#### 设计原则
- 不做单纯聊天框
- 强调“答案 + 证据链”
- 每次回答可溯源到文档与片段

### 5.4 风险信息抽取模块

#### 功能
从文档片段中抽取风险相关信息，结构化入库，供审核与报告模块复用。

#### 输出特点
采用结构化抽取，不采用无约束自然语言结果。

#### 典型字段
- 公司名称
- 风险类型
- 风险等级
- 时间
- 证据原文
- 摘要
- 置信度
- 审核状态

#### 页面能力
- 发起抽取
- 查看抽取结果
- 人工审核
- 按公司/类型/时间过滤

### 5.5 风险报告模块

#### 功能
基于已抽取的风险事件进行聚合，生成可视化报告与文本总结。

#### 支持方式
- 按公司生成
- 按时间区间生成

#### 报告内容
- 公司信息
- 风险事件列表
- 高频风险类型统计
- 模型生成总结
- 引用来源

### 5.6 模型与评测模块

#### 功能
- 模型列表展示
- 本地模型切换
- Embedding 模型配置
- 提示词模板配置
- RAG 参数配置
- 评测结果记录与展示

#### 设计目的
为第二阶段模型接入和 LoRA 微调评估提供统一入口。

## 6. 技术方案设计

### 6.1 后端技术选型
- Django：业务主框架
- Django REST Framework：REST API
- Celery：异步任务
- Redis：缓存与消息队列
- MySQL：结构化业务数据存储

### 6.2 AI 与知识检索技术
- LangChain：问答、检索、抽取链路编排
- Ollama：本地模型统一接入
- Milvus：向量索引管理
- Embedding 模型：后续可配置为本地 Embedding 服务

### 6.3 设计理由
1. Django 适合快速构建后台管理和清晰业务边界。
2. Vue 适合前端页面组件化组织与展示。
3. Celery + Redis 适合处理解析、抽取、报告生成等耗时任务。
4. LangChain 适合快速构建 RAG 与结构化抽取链。
5. Milvus 适合独立管理向量索引，避免向量与业务表混杂。
6. Ollama 适合第一阶段快速接入本地模型。

## 7. 关键业务链路设计

### 7.1 文档入库链路

```text
上传文档
 -> 保存文件
 -> 创建 document 记录
 -> 提交 ingestion_task
 -> 文档解析
 -> 文本清洗
 -> 文本切块
 -> 生成 embedding
 -> 写入 Milvus
 -> 保存 chunk 与 vector_id 映射
 -> 更新状态
```

### 7.2 问答链路

```text
用户提问
 -> query 预处理
 -> 混合召回（关键词 + 向量）
 -> 结果重排/过滤
 -> 选取 Top-K 证据
 -> 大模型生成答案
 -> 返回答案 + 来源 + 原文片段
 -> 保存 chat_message 与 retrieval_log
```

### 7.3 风险抽取链路

```text
选择文档
 -> 遍历 chunk
 -> 调用结构化抽取链
 -> 生成 risk_event JSON
 -> 批量落库
 -> 人工审核
 -> 进入报告聚合
```

### 7.4 风险报告链路

```text
选择公司或时间区间
 -> 读取 risk_event
 -> 聚合统计
 -> 生成摘要 Prompt
 -> 输出总结
 -> 保存 report
```

## 8. RAG 与抽取策略设计

### 8.1 第一版采用 2-step RAG
第一版只采用固定检索前置的 2-step RAG，不使用复杂 Agent。原因如下：

1. 业务流程更稳定，便于演示。
2. 检索行为可控，结果更容易调优。
3. 响应时间可预测，适合毕业设计实现。
4. 便于后续对比“是否使用 RAG”的效果差异。

### 8.2 检索策略
建议采用混合检索：

- 关键词检索：适合公司名、财务指标、风险术语等精确匹配
- 向量检索：适合语义相近段落召回
- 重排：提高最终命中文本质量

### 8.3 片段设计建议
切块建议同时保留：

- `document_id`
- `page_no`
- `chunk_index`
- `company_name`
- `report_period`
- `source_type`

后续问答与抽取都依赖这些元数据实现溯源。

### 8.4 风险抽取策略
风险抽取建议拆成以下步骤：

1. 文本预筛选：减少明显无关片段
2. 结构化字段抽取：输出 JSON
3. 落库去重：避免同一片段重复事件
4. 人工审核：保障结果可信度

## 15. 模型抽象层设计

### 15.1 抽象目标
避免把模型调用逻辑直接写死在业务接口中。

### 15.2 建议接口

```python
class BaseLLMProvider:
    def chat(self, messages, **kwargs):
        raise NotImplementedError

class BaseEmbeddingProvider:
    def embed_documents(self, texts):
        raise NotImplementedError

class BaseRetriever:
    def search(self, query, kb_id, top_k=5):
        raise NotImplementedError
```

### 15.3 适配原则
- 业务服务只依赖抽象接口
- Ollama、微调模型、本地 Embedding 都通过适配器接入
- 切换模型不影响上层 API 与页面

## 19. 项目风险与应对

### 风险 1：PDF 解析质量不稳定
应对：第一版优先保证常见年报 PDF，可对异常文档做失败提示与重传。

### 风险 2：问答结果不稳定
应对：强化引用来源展示，限制回答范围，只基于召回片段作答。

### 风险 3：抽取结果噪声大
应对：结构化抽取 + 去重 + 人工审核。

### 风险 4：开发周期紧张
应对：先完成平台闭环，再补模型训练实验。

## 20. 结论

本设计将系统拆分为平台层、AI 能力层、模型层三部分，采用“平台优先、模型增强后置”的实现路线，优先落地一个可演示、可扩展、可评估的金融风控领域大模型应用平台。第一版重点解决知识入库、检索增强问答、风险信息结构化抽取和风险报告生成问题；第二阶段再通过 LoRA 微调与本地模型接入增强领域能力，从而形成完整的毕业设计成果链路。
