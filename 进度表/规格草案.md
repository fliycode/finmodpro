# 面向金融风控领域的领域大模型应用平台设计说明

日期：2026-03-28

## 1. 项目定位

本项目面向“通用大模型在金融领域知识不足、易产生幻觉”的问题，构建一个以金融年报、研报、新闻为主要知识源的领域大模型应用平台。平台采用 Vue + Django 的前后端分离架构，使用 LangChain 组织 RAG 与结构化抽取链路，使用 Redis + Celery 处理异步任务，使用 MySQL 存储业务数据，使用 Milvus 管理向量索引，并通过 Ollama 接入本地大模型。

平台目标不是先追求复杂的多智能体系统，而是优先构建稳定、可追溯、可验证的金融风控应用闭环，具体包括：

1. 金融知识库构建与管理；
2. 基于知识库的智能问答；
3. 风险信息抽取与结构化落库；
4. 风险报告生成与展示；
5. 本地模型接入与后续 LoRA 微调扩展。

## 2. 设计原则

1. 先平台后模型：优先完成平台主链路，再接入本地模型与微调实验。
2. 先 2-step RAG 后 Agent：第一期使用固定检索 -> 生成链路，保证稳定和可控。
3. 问答与抽取分链路：问答负责检索增强回答，抽取负责结构化事件识别。
4. 可追溯优先：所有回答和抽取结果必须保留来源片段与证据文本。
5. 易扩展：模型、Embedding、向量库、提示词通过配置和适配层解耦。

## 3. 总体架构

### 3.1 前端层

Vue 前端负责页面展示和用户交互，核心页面包括：
- 登录与用户中心
- 知识库管理
- 文档中心
- 智能问答
- 风险抽取
- 风险报告
- 模型与评测

### 3.2 API 层

Django + DRF 提供 REST API，负责：
- 登录、认证、权限控制
- 知识库、文档、任务状态管理
- 会话与消息管理
- 风险事件与报告管理
- 模型配置与评测记录管理

### 3.3 AI 服务层

独立封装 AI 能力，包含：
- 文档解析服务
- 文本切块服务
- 向量化服务
- 检索服务
- 问答服务
- 风险抽取服务
- 报告生成服务

### 3.4 异步任务层

使用 Celery + Redis 完成：
- 文档入库
- 切块与向量化
- 索引重建
- 批量抽取
- 批量报告生成

### 3.5 数据层

- MySQL：业务主库
- Redis：缓存、队列、任务状态
- Milvus：向量库
- 文件存储：原始 PDF/文档附件
- Ollama：本地模型服务

## 4. 功能模块设计

### 4.1 用户与权限模块

角色建议控制为两类：
- 管理员：admin
- 分析师：analyst

第一版只做页面级和接口级权限，不做复杂 RBAC。

### 4.2 知识库管理模块

功能流程：
1. 新建知识库
2. 上传文档
3. 触发解析
4. 文本切块
5. 生成向量
6. 写入向量库
7. 查看任务状态与失败原因

知识源优先支持：
- 金融年报 PDF
- 研报 PDF / TXT
- 新闻文本

### 4.3 智能问答模块

输入：用户问题 + 当前知识库

输出：
- 回答内容
- 命中的来源文档
- 引用片段
- 检索得分或置信提示

核心链路：
用户问题 -> Query 预处理 -> 检索 -> 重排 -> 生成答案 -> 返回答案与来源

### 4.4 风险信息抽取模块

采用结构化输出，将抽取结果统一为风险事件表。建议字段：
- company_name
- report_period
- risk_type
- risk_level
- event_time
- summary
- evidence_text
- source_doc_id
- source_chunk_id
- confidence
- review_status

处理流程：
文档切块 -> 批量调用抽取链 -> 输出结构化 JSON -> 落库 -> 人工审核

### 4.5 风险报告模块

基于风险事件表进行聚合，而不是直接对整篇文档做长文本摘要。

支持：
- 按公司生成风险报告
- 按时间区间生成风险摘要

报告内容：
- 公司信息
- 风险事件列表
- 风险类别分布
- 模型总结
- 引用来源

### 4.6 模型与评测模块

功能包括：
- 模型配置管理
- Embedding 模型配置
- 检索参数配置
- Prompt 模板配置
- 评测集管理
- 评测记录展示

## 6. 向量库与元数据设计

Milvus 负责存储向量和检索，不直接承担业务主数据。

建议 collection 设计字段：
- pk / vector_id
- document_id
- chunk_id
- kb_id
- company_name
- report_period
- doc_type
- page_no
- text
- dense_vector
- sparse_vector（第二阶段可选）

其中：
- MySQL 保存业务主键与元数据；
- Milvus 保存向量与检索过滤字段；
- 通过 vector_id / chunk_id 建立映射关系。

## 7. RAG 与抽取链路设计

### 7.1 问答链

用户问题
-> 问题预处理
-> 关键词召回 + 向量召回
-> 结果合并 / 重排
-> Top-K 片段拼接
-> 大模型生成答案
-> 返回答案 + 来源 + 引用片段

### 7.2 风险抽取链

选择文档
-> 遍历切块
-> 调用结构化抽取链
-> 输出风险事件 JSON
-> 保存到 risk_event
-> 人工审核
-> 报告生成

### 7.3 设计原因

- 问答与抽取职责不同，应该分链；
- 问答强调 grounded answer；
- 抽取强调 schema 一致性与落库可用性；
- 报告基于 risk_event 做聚合更容易控制质量。

## 11. 非功能要求

1. 可追溯：所有回答和抽取均需保留来源。
2. 可扩展：模型和向量库支持替换。
3. 可维护：业务 app、AI 能力、异步任务分层。
4. 可演示：支持从上传文档到风险报告的完整闭环。

## 12. 本设计的边界

第一期不做：
- 复杂多智能体编排
- 多租户复杂权限系统
- 实时流式复杂前端交互
- 全自动无人审核的风控结论

平台定位是“金融分析师辅助决策工具”，不是“完全自动决策系统”。
