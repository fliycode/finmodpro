# FinModPro 管理员演示增强设计文档

日期：2026-03-24

## 1. 背景

FinModPro 当前已经具备：

- 基础登录 / 注册能力
- 基于 JWT 的认证闭环
- 基于 Django `Group / Permission` 的 RBAC 第一版
- `GET /api/auth/me` 获取当前用户角色与权限
- 前端基础 RBAC 状态展示

下一步目标不是继续停留在“仅有权限判断”，而是补出更像完整毕设作品的**管理员演示能力**，让 RBAC 从“后端能力”变成“可见、可讲、可操作”的系统演示链路。

## 2. 目标

本阶段目标是构建一套适合毕设演示的管理员后台增强方案，使系统可以完整展示：

1. 普通用户与管理员看到不同界面入口
2. 管理员拥有独立后台入口
3. 管理员可以查看用户列表
4. 管理员可以查看用户当前角色与权限摘要
5. 管理员可以为用户分配多个角色
6. 管理员可以移除用户已有角色
7. 更新完成后前端局部刷新，体现权限变化

## 3. 页面结构方案

本方案采用：

- **管理员仪表盘**
- **用户管理子页面**

而不是把管理能力直接塞进现有登录页面或只做单页面板。

### 为什么这样设计

- 更像正式后台系统
- 更适合毕设演示时讲清楚“普通用户界面”和“管理员界面”的区别
- 页面职责清晰，便于前后端继续演进
- 比单页面板更有系统感，但复杂度仍然可控

## 4. 页面一：管理员仪表盘

管理员仪表盘作为管理员登录后的专属入口页。

### 4.1 主要职责
- 展示当前登录管理员身份
- 展示当前用户的角色与权限摘要
- 展示系统概览信息
- 提供进入用户管理页的入口
- 强化 RBAC 演示效果

### 4.2 页面模块

#### A. 当前身份卡片
展示：
- 用户名
- 邮箱
- 当前角色列表
- 权限数量
- 管理员身份标记

#### B. 系统概览卡片
展示：
- 用户总数
- 角色总数
- 当前管理员可管理权限数

这些指标可通过现有接口聚合得到，用于增强后台感和答辩展示效果。

#### C. 管理入口区
至少提供：
- “进入用户管理”按钮

#### D. RBAC 演示区
强调：
- 普通用户不可见后台入口
- 管理员可见后台入口
- 当前用户具备的角色和权限决定可访问功能

## 5. 页面二：用户管理页

用户管理页是本阶段管理员演示增强的核心页面。

### 5.1 页面职责
- 展示用户列表
- 支持搜索与筛选
- 展示当前角色与权限摘要
- 提供角色编辑能力

### 5.2 顶部操作区
建议包含：
- 用户名 / 邮箱搜索框
- 角色筛选器
- 刷新按钮

### 5.3 用户表格字段
本阶段采用接近后台版展示，字段包括：

- 用户名
- 邮箱
- 角色
- 权限摘要
- 管理员标记
- 注册时间
- 操作列

### 5.4 操作区
每行提供：
- “编辑角色”按钮

### 5.5 角色编辑交互
采用：
- **右侧抽屉** 或 **弹窗**

内容包括：
- 当前用户基础信息
- 当前已有角色
- 全部可选角色
- 多选切换
- 保存 / 取消按钮

### 5.6 分配规则
本阶段明确支持：
- 为用户分配多个角色
- 移除用户已有角色

即：
- 用户角色关系继续保持多角色模型
- 前端采用多选交互
- 后端采用整集合覆盖式更新

## 6. 后端接口设计

### 6.1 新增接口

#### `GET /api/admin/users`
返回用户列表，供用户管理页展示。

每条记录建议至少包含：

```json
{
  "id": 1,
  "username": "alice",
  "email": "alice@example.com",
  "groups": ["member"],
  "permissions": ["view_dashboard"],
  "is_superuser": false,
  "is_staff": false,
  "date_joined": "2026-03-24T15:00:00+08:00"
}
```

#### `GET /api/admin/groups`
返回所有可分配角色（Group）列表。

示例：
```json
[
  { "id": 1, "name": "super_admin" },
  { "id": 2, "name": "admin" },
  { "id": 3, "name": "member" }
]
```

#### `PUT /api/admin/users/{id}/groups`
更新某个用户的角色集合。

请求体：
```json
{
  "groups": ["admin", "member"]
}
```

### 6.2 为什么采用整集合覆盖更新
本阶段不拆成“单独 add / remove 接口”，原因是：

- 前端逻辑更清晰
- 后端实现更简单
- 演示时更容易说明“管理员提交最新角色集合”
- 更适合多角色场景

## 7. 接口鉴权规则

后台管理接口必须受到 RBAC 保护。

建议：

- `GET /api/admin/users` → 需要 `view_user`
- `GET /api/admin/groups` → 需要 `view_role`
- `PUT /api/admin/users/{id}/groups` → 需要 `assign_role`

这样前后端展示与后端安全边界保持一致。

## 8. 数据流设计

### 8.1 管理员登录后
1. 登录成功
2. 调用 `GET /api/auth/me`
3. 判断是否具备管理员角色/权限
4. 展示管理员入口

### 8.2 进入用户管理页
1. 调用 `GET /api/admin/users`
2. 调用 `GET /api/admin/groups`
3. 渲染用户表格与角色选项

### 8.3 更新角色
1. 打开角色编辑抽屉 / 弹窗
2. 勾选或取消角色
3. 调用 `PUT /api/admin/users/{id}/groups`
4. 成功后局部刷新当前用户行数据

## 9. 前端交互策略

### 9.1 局部刷新
本阶段明确采用：
- **局部刷新**

也就是：
- 更新某个用户角色成功后
- 不整页刷新用户列表
- 只更新当前用户行的数据

这样体验更好，也更像正式后台。

### 9.2 可见性控制
前端继续承担：
- 菜单显示控制
- 页面入口显示控制
- 按钮显示控制

但真正安全校验仍以后端权限检查为准。

### 9.3 视觉策略
本阶段不重新设计一套后台系统视觉风格，而是在现有 FinModPro 风格上延续：

- 卡片式布局
- 圆角与轻量阴影
- 清晰状态标签
- 简洁表格与侧边抽屉/弹窗

目标是：
- 保持统一视觉语言
- 增强系统感
- 不额外引入无关复杂度

## 10. 演示链路

本阶段完整演示链路应为：

1. 普通用户登录
   - 只能看到普通界面
   - 看不到管理员入口

2. 管理员登录
   - 能看到管理员入口
   - 可进入管理员仪表盘

3. 管理员进入用户管理页
   - 可查看所有用户及角色/权限摘要

4. 管理员修改某个用户角色
   - 保存成功后当前用户行数据即时变化
   - 可展示角色变更带来的权限变化

这条链路可以完整支撑毕设答辩时对 RBAC 的展示。

## 11. 本阶段不做

为保持聚焦，本阶段明确不做：

- 多租户 / 多组织 / 多项目后台
- 审计日志系统
- 权限树编辑器
- 菜单可配置平台
- 复杂审批流
- 分页、导出、批量操作等企业级后台功能全量实现

如果需要，可在演示文案中说明这些属于后续扩展方向。

## 12. 验收标准

本阶段完成后，应满足：

1. 存在管理员仪表盘页面
2. 存在用户管理子页面
3. 普通用户不可见管理员入口
4. 管理员可访问后台页面
5. 后台可展示接近后台版的用户表格
6. 管理员可对用户进行多角色增删
7. 保存后支持局部刷新
8. 后端接口有明确权限保护
9. 整体可作为毕设 RBAC 演示链路使用

## 13. 结论

本方案确定以：

- **管理员仪表盘 + 用户管理子页面**
- **多角色增删**
- **接近后台版用户列表**
- **局部刷新**

作为 FinModPro RBAC 第二阶段的演示增强方向。

该方案在当前毕设演示目标下，兼顾：

- 系统感
- 演示完整度
- 实现复杂度
- 继续复用当前 Django Group / Permission 方案
