## 一、论文选题与题目建议

结合 FinModPro 当前实现，较适合的论文题目方向如下：

1. **基于大语言模型与检索增强生成的金融风控平台设计与实现**
2. **面向金融知识管理与风险分析的智能问答系统设计与实现**
3. **融合 RAG 与图增强检索的金融风险控制系统设计与实现**
4. **基于 Vue 与 Django 的金融知识库问答与风险分析平台设计与实现**

如果学校更偏“系统设计与实现”风格，优先使用第 1 或第 3 个题目；如果更偏“信息系统/软件工程”，第 2 或第 4 个题目会更稳。

## 二、项目事实摘要

FinModPro 是一个面向金融风控与金融知识管理场景的前后端分离系统。前端采用 Vue 3、Vite、Vue Router 和 Element Plus 构建，后端采用 Django 5、Django REST Framework 与 Celery 构建，业务数据存储在 MySQL 中，异步任务与缓存依赖 Redis，向量检索由 Milvus 提供，模型网关由 LiteLLM 提供，图增强检索由 LightRAG 提供，当前生产编排中的图数据库后端为 Neo4j，文档解析采用 Unstructured API，并在 PDF 场景下保留 `pypdf` fallback。

系统面向两类主要用户：一类是使用 `/workspace` 的金融分析师，主要完成智能问答、知识检索、风险分析与舆情分析；另一类是使用 `/admin` 的管理员或模型运营人员，主要完成用户权限管理、知识库治理、模型配置、网关观测、成本分析、评测与 LightRAG 图谱治理。

当前代码已经落地的主功能包括：

- 用户注册、登录、退出、JWT 鉴权、refresh cookie 会话刷新
- 基于 Django Group / Permission 的 RBAC
- 金融文档上传、数据集管理、文档版本、chunk 查看、单文档/批量 ingest
- RAG 检索、SSE 流式问答、会话与记忆管理、引用依据返回
- 风险事件抽取、风险审核、公司/时间区间风险报告生成、舆情分析
- LLM 模型配置、Prompt 配置、LiteLLM 网关观测/成本/错误追踪、评测、微调任务管理
- 健康检查、统计看板、审计日志

## 三、写论文时必须遵守的事实边界

以下内容可以写进论文：

1. 系统使用 Vue 3 + Django 5 的前后端分离架构。
2. 系统包含工作区与管理台两个主要端面。
3. 系统具备知识库、问答、风险分析、模型中台、审计与健康检查等模块。
4. 系统使用 Milvus 做向量检索，使用 LiteLLM 做模型统一接入。
5. 系统已接入 LightRAG 管理入口与图增强检索链路。

以下内容在写作时需要分清口径：

1. **可以写当前生产环境已接入 Neo4j。** `docker-compose.prod.yml` 已显式编排 `neo4j` 服务，且 LightRAG 图存储配置为 `Neo4JStorage`。
2. **但要区分生产事实与默认回退配置。** `backend/config/settings.py` 在未注入环境变量时仍保留 `LIGHTRAG_GRAPH_STORAGE=NetworkXStorage` 默认值，因此若论文提到本地默认配置，需要注明这是回退设置而非生产部署口径。
3. **不要写 access token 存在 localStorage。** 当前前端 access token 是内存态，refresh 依赖 HTTP-only cookie。
4. **不要写默认 SQLite 是当前正式数据库。** 代码里的 Django 设置要求 MySQL。
5. **不要把历史文档中的规划能力全部写成“已完成”。** 论文中应区分“已实现”“设计支持”“后续扩展”。

## 四、按论文模板映射 FinModPro 的写法

### 4.1 第一章 前言

这一章重点是把系统的背景、问题、研究内容和论文结构讲清楚。

#### 1.1 研究背景及意义

可围绕以下逻辑展开：

- 金融行业文档多、信息更新快、风险识别对时效性要求高
- 传统人工研判成本高，普通关键词检索无法很好支持跨文档、多证据链分析
- 大语言模型虽强，但直接使用存在幻觉、依据不透明、知识不可控的问题
- 因此需要把 **知识库、RAG、风险抽取、模型治理** 结合起来，形成可落地的平台

可用的项目化表达：

> 随着金融市场信息规模持续扩大，上市公司公告、研报、舆情资讯和内部风险文档不断累积，金融机构在风险识别、知识管理和决策支持方面面临更高要求。传统依赖人工阅读与关键词检索的方式，存在检索效率低、跨文档证据关联能力弱、风险发现滞后等问题，难以满足实际业务需要。大语言模型的发展为金融知识问答与风险分析提供了新的思路，但单纯依赖通用模型仍存在知识时效性不足、回答依据不透明和结果可控性不强等局限。因此，研究并实现一个融合知识库、检索增强生成和风险分析能力的金融风控平台，具有较强的理论意义和工程价值。

#### 1.2 国内外发展及研究现状

建议从三条线写：

1. 智能问答与 RAG
2. 金融风控信息化系统
3. 图增强检索 / 知识图谱问答

结尾可以收束为：

> 现有研究虽已在问答生成、向量检索和风险识别等方面取得一定成果，但在金融场景下仍存在知识管理与模型治理割裂、问答依据链不足、图关系分析能力不强等问题。基于此，本文结合 Vue、Django、Milvus、LiteLLM 和 LightRAG 等技术，设计并实现了一个面向金融风控场景的智能平台。

#### 1.3 研究内容

这里直接写“本文做了什么”：

- 设计并实现前后端分离金融风控平台
- 实现认证与 RBAC
- 实现知识库文档入库与向量检索
- 实现基于 LangGraph 的问答链路
- 实现风险事件抽取与报告
- 实现模型中台与运维观测

#### 1.4 论文结构

可直接采用模板句式：

> 本文共分为七章。第一章介绍研究背景、研究意义及论文结构；第二章介绍系统开发所涉及的相关技术；第三章对系统需求进行分析；第四章给出系统总体设计、模块设计与数据库设计；第五章介绍系统主要功能的实现过程；第六章对系统进行测试并分析结果；第七章对全文进行总结，并对后续优化方向进行展望。
