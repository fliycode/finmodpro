# FinModPro 论文撰写参考文档（代码事实版）

> 用途：给更专业的模型或论文写作助手提供 **FinModPro 的论文素材与章节映射**。
> 使用原则：**代码事实优先，禁止把历史规划当成已实现功能写进论文。**

## 一、论文选题与题目建议

结合 FinModPro 当前实现，较适合的论文题目方向如下：

1. **基于大语言模型与检索增强生成的金融风控平台设计与实现**
2. **面向金融知识管理与风险分析的智能问答系统设计与实现**
3. **融合 RAG 与向量检索的金融风险控制系统设计与实现**
4. **基于 Vue 与 Django 的金融知识库问答与风险分析平台设计与实现**

如果学校更偏"系统设计与实现"风格，优先使用第 1 或第 3 个题目；如果更偏"信息系统/软件工程"，第 2 或第 4 个题目会更稳。

## 二、项目事实摘要

FinModPro 是一个面向金融风控与金融知识管理场景的前后端分离系统。前端采用 Vue 3、Vite、Vue Router 和 Element Plus 构建，后端采用 Django 5、Django REST Framework 与 Celery 构建，业务数据存储在 MySQL 中，异步任务与缓存依赖 Redis，向量检索由 Milvus 提供，检索抽象层使用 LlamaIndex，模型调用通过 OpenAI 兼容 provider 直连。文档解析方面，PDF 主解析由 pymupdf4llm 承担，后端在解析失败时保留 `pypdf` fallback，DOCX 使用 python-docx 解析。

系统面向两类主要用户：一类是使用 `/workspace` 的金融分析师，主要完成智能问答、知识检索、风险分析；另一类是使用 `/admin` 的管理员或模型运营人员，主要完成用户权限管理、知识库治理、模型配置、网关观测、成本分析、评测、系统监控与告警。

当前代码已经落地的主功能包括：

- 用户注册、登录、退出、JWT 鉴权、refresh cookie 会话刷新
- 基于 Django Group / Permission 的 RBAC
- 金融文档上传、数据集管理、文档版本、chunk 查看、单文档/批量 ingest
- LlamaIndex-over-Milvus RAG 检索、SSE 流式问答、会话与记忆管理、引用依据返回
- 结构化风险事件抽取、裁决丰富、验证、审核、风险报告生成（xlsx 导出）、情感分析 API
- LLM 模型配置、Prompt 配置、网关观测/成本/错误追踪、评测、微调任务管理
- 健康检查、统计看板、审计日志、系统监控与告警管理

## 三、写论文时必须遵守的事实边界

以下内容可以写进论文：

1. 系统使用 Vue 3 + Django 5 的前后端分离架构。
2. 系统包含工作区与管理台两个主要端面。
3. 系统具备知识库、问答、风险分析、模型中台、审计与健康检查等模块。
4. 系统使用 Milvus 做向量检索，使用 LlamaIndex 做检索抽象层。
5. 系统使用 LangGraph 做问答流程编排。

以下内容在写作时需要分清口径：

1. **不要写 access token 存在 localStorage。** 当前前端 access token 是内存态，refresh 依赖 HTTP-only cookie。
2. **不要写默认 SQLite 是当前正式数据库。** 代码里的 Django 设置要求 MySQL。
3. **不要把历史文档中的规划能力全部写成"已完成"。** 论文中应区分"已实现""设计支持""后续扩展"。
4. **系统没有 LiteLLM 网关、LightRAG 图检索、Neo4j 图数据库。** 这些技术之前被评估过但已移除。

## 四、按论文模板映射 FinModPro 的写法

### 4.1 第一章 前言

这一章重点是把系统的背景、问题、研究内容和论文结构讲清楚。

#### 1.1 研究背景及意义

可围绕以下逻辑展开：

- 金融行业文档多、信息更新快、风险识别对时效性要求高
- 传统人工研判成本高，普通关键词检索无法很好支持跨文档、多证据链分析
- 大语言模型虽强，但直接使用存在幻觉、依据不透明、知识不可控的问题
- 因此需要把 **知识库、RAG、风险抽取、模型治理** 结合起来，形成可落地的平台

可用的项目化表达：

> 随着金融市场信息规模持续扩大，上市公司公告、研报、舆情资讯和内部风险文档不断累积，金融机构在风险识别、知识管理和决策支持方面面临更高要求。传统依赖人工阅读与关键词检索的方式，存在检索效率低、跨文档证据关联能力弱、风险发现滞后等问题，难以满足实际业务需要。大语言模型的发展为金融知识问答与风险分析提供了新的思路，但单纯依赖通用模型仍存在知识时效性不足、回答依据不透明和结果可控性不强等局限。因此，研究并实现一个融合知识库、检索增强生成和风险分析能力的金融风控平台，具有较强的理论意义和工程价值。

#### 1.2 国内外发展及研究现状

建议从三条线写：

1. 智能问答与 RAG
2. 金融风控信息化系统
3. 向量检索与语义搜索

结尾可以收束为：

> 现有研究虽已在问答生成、向量检索和风险识别等方面取得一定成果，但在金融场景下仍存在知识管理与模型治理割裂、问答依据链不足等问题。基于此，本文结合 Vue、Django、Milvus、LlamaIndex 和 LangGraph 等技术，设计并实现了一个面向金融风控场景的智能平台。

#### 1.3 研究内容

这里直接写"本文做了什么"：

- 设计并实现前后端分离金融风控平台
- 实现认证与 RBAC
- 实现知识库文档入库与向量检索
- 实现基于 LangGraph 的问答链路
- 实现风险事件抽取与报告
- 实现模型中台与运维观测

### 4.2 第二章 相关知识和技术

这一章建议拆成如下小节：

#### 2.1 Vue 3 前端框架
说明 Vue 3 的组件化、响应式、适合构建单页应用，并补一句"本文用于工作区和管理台前端实现"。

#### 2.2 Vite 构建工具
说明其开发启动快、构建效率高，适合前后端分离项目。

#### 2.3 Vue Router 路由技术
说明其用于工作区和管理台路由组织、权限控制与页面跳转。

#### 2.4 Django 与 Django REST Framework
说明 Django 适合快速构建后台系统，DRF 便于提供 REST API。

#### 2.5 MySQL 与 Redis
说明 MySQL 负责业务持久化，Redis 负责缓存和 Celery 异步任务。

#### 2.6 Milvus 向量数据库
说明其用于文档 chunk 向量检索，支撑 RAG。

#### 2.7 LlamaIndex 检索框架
说明其作为 Milvus 的上层抽象，统一向量存储、嵌入和检索接口，支持混合检索（全文 + 向量 + RRF 融合）。

#### 2.8 LangGraph 问答编排
说明其用于问题路由、检索、上下文构建和问答流程组织。

#### 2.9 LLaMA-Factory 微调框架

可重点说明以下内容：

- LLaMA-Factory 是面向大语言模型的高效微调框架，支持 SFT（监督微调）和 LoRA（低秩适配）等多种策略。
- 在本文系统中，LLaMA-Factory 作为外部微调执行引擎，接收平台导出的训练数据，完成训练后通过回调接口将结果回写平台。
- 平台内通过 runner agent（FastAPI 服务）桥接 LLaMA-Factory 训练调度与状态同步。

#### 2.10 pymupdf4llm PDF 解析技术
可重点说明以下内容：

- pymupdf4llm 是基于 PyMuPDF 的 Markdown 格式 PDF 转写库，能够按页访问文档内容，并输出结构化 Markdown。
- 与只做纯文本抽取的方案相比，pymupdf4llm 更适合保留金融 PDF 中的结构线索（标题、表格、段落），便于后续生成更稳定的 element 和页码元数据。
- 在本文系统中，pymupdf4llm 被用于 PDF 主解析路径，负责将 PDF 文档转为结构化 element 输出。

#### 2.11 解析容错与 `pypdf` fallback
可重点说明以下内容：

- 当 pymupdf4llm 解析异常时，后端会通过 `pypdf` 执行 PDF 文本回退提取，从而保证知识库入库流程具备基本容错能力。
- 这种设计兼顾了解析质量与工程稳定性：主路径提升 PDF 提取质量，回退路径保证系统在异常时仍可继续工作。
