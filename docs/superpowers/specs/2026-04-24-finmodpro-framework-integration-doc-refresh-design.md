# FinModPro 开源框架接入文档重构设计

日期：2026-04-24

## 1. 目标

将现有的 `docs/2026-04-22-finmodpro-open-source-framework-integration-plan.md` 从“阶段性建议稿”重构为一套更稳定的文档体系，供项目维护者在后续判断开源框架是否继续接入、如何接入、是否允许进入当前生产环境时直接使用。

这次设计的目标不是扩写更多背景说明，而是把文档体系拆成一个总决策入口和两份配套规范，使“为什么这样决策”和“在什么条件下才能推进”分开承载，避免主文档继续混入过多细节后再次失真。

## 2. 当前事实

当前仓库和部署现实已经足以支持一次文档重构：

- 当前服务器就是生产环境，部署方式是单机 `docker compose`，由服务器本地执行 `scripts/deploy.sh`
- 当前不存在 staging、blue-green、canary 或数据库级自动回滚
- 代码中已经存在若干接入边界：
  - `UNSTRUCTURED_API_URL` 外部解析边界
  - `common/observability.py` 的 Langfuse 业务埋点包装
  - `FineTuneRun`、`runner-spec`、`callback` 的训练控制面接口
  - `runtime_service` 中的 `DeepSeek / DashScope / LiteLLM` provider 选择逻辑
- 当前文档的主要问题不是缺少观点，而是混合了：
  - 架构决策
  - 生产约束
  - 组件细节
  - 阶段计划
  - 定性验收结论

这导致它适合“阅读理解”，不够适合“后续取舍判断”。

## 3. 范围

### 3.1 In Scope

- 重写主文档为总决策入口文档
- 新增两份配套 spec，分别承接生产环境约束和组件接入边界
- 用当前仓库、部署脚本、现有设计文档与官方文档核对结论重建文档依据
- 明确每个组件的当前结论、推荐接法、禁区与重新评估条件
- 保留分阶段建议，但改为“进入条件 / 推进门槛”，而不是口号式路线图

### 3.2 Out of Scope

- 重写或扩展现有代码实现
- 新增部署系统、备份系统或 staging 环境
- 为每个框架编写完整实施手册
- 深入拉取和研究大型上游源码仓库
- 为当前文档体系补全所有未来可能需要的细分规范

## 4. 设计原则

- **主文档只做总决策**：回答“该不该做、为什么、做到哪一步”
- **生产约束单独承载**：回答“当前生产环境允许什么样的变更”
- **组件边界单独承载**：回答“每个组件到底怎么接、做到什么算完成”
- **基于证据，不基于愿景**：优先采用当前仓库与部署现实，外部官方文档只用于校正组件定位与边界
- **整体结论从严不从宽**：在单机生产现实下，默认保守，优先外部化、旁路化、可回退

## 5. 文档体系设计

### 5.1 主文档

保留现有路径：

- `docs/2026-04-22-finmodpro-open-source-framework-integration-plan.md`

但文档角色改为：

- FinModPro 当前开源框架接入的总决策文档
- 面向项目维护者，用于后续取舍和阶段判断
- 不承担实现细节和运维执行细则

建议重写后的章节结构：

1. 文档定位与适用范围
2. 当前事实基线
3. 总体决策摘要
4. 总原则
5. 分组件结论矩阵
6. 分阶段进入条件
7. 当前明确不做的事项
8. 配套 spec 索引

### 5.2 配套 spec A：生产环境变更约束与执行门槛

建议新增：

- `docs/superpowers/specs/2026-04-24-finmodpro-production-integration-guardrails.md`

职责：

- 说明当前服务器就是生产环境这一前提下的准入门槛
- 给出框架接入的风险分级和最小变更原则
- 明确何种变更可以在当前环境推进，何种变更必须迁出或延后

重点内容：

- 当前部署模型与回滚边界
- 对 compose 变更、状态服务新增、镜像膨胀、网络依赖增加的判断口径
- 低风险 / 中风险 / 禁止类接入分类
- 最小回退要求与最低验证要求

### 5.3 配套 spec B：组件接入边界与验收口径

建议新增：

- `docs/superpowers/specs/2026-04-24-finmodpro-framework-integration-boundaries.md`

职责：

- 为 `Unstructured / Langfuse / LLaMA-Factory / LiteLLM` 提供统一的接入边界、进入条件、验收口径和降级条件
- 避免主文档重新膨胀成“组件细节大全”

重点内容：

- 每个组件的角色定位
- 推荐接法与不推荐接法
- 与当前代码/部署的耦合提醒
- 最低验收标准
- 进入条件与退出条件

## 6. 证据来源与使用方式

本次文档重构采用以下证据来源，优先级从高到低如下：

1. 当前仓库代码、配置和部署脚本
2. 当前仓库中的既有设计文档
3. 近期提交记录
4. 各组件官方文档

外部官方文档本次只用于校正组件定位与部署边界，不用于替代对 FinModPro 自身现实的判断。

本轮已核对的官方结论包括：

- **LiteLLM**：官方文档将 proxy 作为独立部署对象，并注明生产至少需要 `4 CPU / 8GB RAM`
- **Langfuse**：官方文档将 Docker Compose 视为低规模/测试方案，生产高可用建议 Kubernetes / Terraform，并依赖 `Postgres + ClickHouse + Redis/Valkey + Blob Storage`
- **Unstructured**：官方文档明确开源库不面向生产场景，生产更偏向 API / VPC / 托管形态
- **LLaMA-Factory**：官方文档和 README 延续其作为 CLI / Web UI / 本地或云训练执行面的定位，适合作为外部训练子系统而非 Django 内嵌能力

## 7. 对主文档的重写策略

主文档重写时应进行以下收口：

- 去掉“像实施方案但又不够细”的段落
- 减少定性口号，增加明确判断
- 把“服务器约束”从背景信息提升为决策前提
- 把 `P0-P4` 从推进愿景改写成进入条件
- 把 “LiteLLM 可选”“Unstructured 按需启用”“Langfuse Cloud 优先”等结论与当前部署现实对齐

特别是以下问题必须在重写时显式修正：

- LiteLLM 在代码层“可选”和在部署层“当前仍在生产栈中”之间的落差
- Unstructured“按需启用”缺少当前生产准入条件的问题
- Langfuse Cloud 的成本、网络与数据边界未被明确写出的问题
- LLaMA-Factory 的 control plane 与推理部署之间仍存在实施责任空白的问题

## 8. 验收标准

当本次文档重构完成时，应满足以下标准：

- 主文档能在 5 分钟内回答当前是否值得继续推进某个框架接入
- 主文档不再承担过多实现细节，且能正确索引到两个配套 spec
- 配套 spec A 能回答“当前生产环境允许什么样的接入变更”
- 配套 spec B 能回答“每个组件怎么接、何时可进下一阶段、做到什么算完成”
- 三份文档之间没有明显冲突、重复定义或范围重叠失控

## 9. 实施顺序

本次文档工作按以下顺序进行：

1. 重写总文档结构与核心决策表达
2. 新增生产环境变更约束 spec
3. 新增组件接入边界与验收口径 spec
4. 回到总文档补齐索引和引用关系
5. 自检范围、冲突、歧义与占位符

## 10. 非目标与边界提醒

这次文档重构不会把 FinModPro 变成一个“框架研究报告”仓库。

文档的核心目的仍然是：在当前生产现实下，帮助维护者做出更稳的接入判断。

因此如果某个上游框架的官方能力与当前系统现实发生冲突，文档应优先忠实反映 FinModPro 的现实约束，而不是追随上游项目的理想部署方式。
