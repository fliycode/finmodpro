# FinModPro LLM 中台与 LLaMA-Factory 管理后台设计

日期：2026-04-19

## 1. 目标

为 FinModPro 的管理后台补齐一组能够明确表达“LLM 基础设施已经接入并正在运行”的页面结构，使管理员能够直观看到：

- 当前平台已经接入了哪些 LLM 基础设施
- 当前启用的模型、路由和解析链路是什么
- 调用、观测、知识库解析和微调链路的运行状态如何
- LLaMA-Factory 任务当前处于哪个阶段，失败点在哪里，产物是否已经回流到 LiteLLM

这次设计的重点不是再做一个“模型配置表单”，而是把现有后端能力组织成一套有辨识度的 **LLM 中台**。

## 2. 当前问题

当前仓库里的后端接入已经具备实际能力：

- LiteLLM 已作为统一模型入口接入
- Langfuse 已作为业务 trace / span 能力接入
- Unstructured 已进入知识库解析链路
- LLaMA-Factory 已完成仓库内第一阶段控制面、runner 契约和 LiteLLM alias 回流

但前端信息架构仍然让这些接入几乎不可感知：

- 管理员能看到的主要入口仍是“模型配置”和“评测结果”
- “模型配置”页承担了过多职责，既像 provider CRUD，又承载微调登记
- 当前页面更像运维表单，而不像“LLM 系统已经接入并在运行”的控制台
- Langfuse、Unstructured、LLaMA-Factory 没有形成清晰的独立认知入口

因此管理员即使进入后台，也很难一眼理解系统已经接了什么、当前是否健康、问题出在哪里。

## 3. 设计范围

### 3.1 In Scope

- 在管理员后台新增一个独立的 **LLM 中台** 信息架构
- 将现有“模型配置”能力收束进 LLM 中台体系
- 新增 LLM 中台总览页
- 新增独立的 LLaMA-Factory 页面
- 为 LiteLLM / Langfuse / Unstructured / LLaMA-Factory 提供明确的状态表达
- 在页面层明确空态、错误态、权限态和加载态
- 保持现有管理员工作流不被破坏

### 3.2 Out of Scope

- 为普通用户增加新的工作区页面
- 在 FinModPro 内完整重建 Langfuse 的 trace 浏览器
- 在 FinModPro 内完整重建 LiteLLM Proxy 的全部配置能力
- 在页面中直接执行真实训练编排或部署编排
- 新增面向终端用户的市场化“AI 产品页”
- 重做现有工作区路由体系

## 4. 设计原则

- **先让系统可见，再让配置可改。**
- **每个页面只回答一个核心问题。**
- **总览页展示运行态，子页承担操作面。**
- **用克制的金融中后台风格表达复杂状态，不做花哨 dashboard。**
- **优先暴露真实链路和失败语义，而不是堆装饰性图表。**
- **保留现有管理员壳层、侧边栏、顶栏和卡片规范。**

## 5. 信息架构

### 5.1 顶层结构

保留现有的两大一级区域：

- 工作区
- 管理台

在“管理台”下新增 **LLM 中台分组**，用于统一承载模型、观测、解析和微调链路。

### 5.2 管理台导航结构

建议结构如下：

- 管理台
  - 仪表盘
  - 用户管理
  - LLM 中台
    - 总览
    - 模型配置
    - 观测
    - 知识库接入
    - LLaMA-Factory
  - 评测结果

说明：

- “评测结果”暂不并入 LLM 中台分组内部，而是继续作为管理台的独立能力存在
- 但 LLM 中台总览页、模型配置页和 LLaMA-Factory 页应提供指向“评测结果”的跳转入口
- 现有 `/admin/models` 需要收束为 LLM 中台下的模型配置页面，而不是孤立入口

### 5.3 路由建议

- `/admin/llm`
- `/admin/llm/models`
- `/admin/llm/observability`
- `/admin/llm/knowledge`
- `/admin/llm/fine-tunes`

旧入口可保留兼容跳转：

- `/admin/models` -> `/admin/llm/models`

## 6. 页面设计

### 6.1 `/admin/llm` 总览页

#### 页面目标

让管理员一进入页面就能知道：

1. 平台到底接了哪些 LLM 基础设施
2. 哪些是健康的，哪些有异常
3. 当前主链路在使用什么模型和入口
4. 最近运行链路发生了什么

#### 页面模块

1. **基础设施状态卡**
   - LiteLLM
   - Langfuse
   - Unstructured
   - LLaMA-Factory

   每张卡显示：

   - 接入状态
   - 最近活跃时间
   - 当前关键状态文案
   - 快速跳转入口

2. **当前模型与链路关系卡**
   - 当前启用聊天模型
   - 当前启用向量模型
   - 当前 provider
   - 当前 endpoint / alias
   - 聊天链路与 embedding 链路的入口说明

3. **近期运行摘要**
   - 最近调用成功 / 失败概览
   - 最近知识库解析摘要
   - 最近微调任务摘要
   - 最近需要处理的异常项

4. **快捷操作区**
   - 去模型配置
   - 去观测
   - 去知识库接入
   - 去 LLaMA-Factory
   - 去评测结果

#### 视觉定位

- 不是首页 hero
- 不是大图表仪表盘
- 是以状态卡、摘要卡、表格摘要为主的密集中后台

### 6.2 `/admin/llm/models` 模型配置页

#### 页面目标

集中管理模型 provider、endpoint、启用关系和基础连通性测试。

#### 页面模块

1. **对话模型配置**
2. **向量模型配置**
3. **当前启用模型摘要**
4. **连接测试与保存动作**
5. **与微调回流模型的关系提示**

#### 调整重点

- 保留现有模型 CRUD 能力
- 从“泛后台配置页”升级为“LLM 中台的模型与路由页”
- 明确区分：
  - 原生 provider
  - LiteLLM alias
  - 微调回流产生的候选模型

### 6.3 `/admin/llm/observability` 观测页

#### 页面目标

让管理员看清“调用是否发生”“失败是否增加”“问题应去哪里排查”。

#### 页面模块

1. **调用摘要**
   - 最近请求数
   - 最近失败数
   - 最近延迟水平

2. **业务链路摘要**
   - Chat
   - Retrieval
   - Risk extraction
   - Sentiment
   - Ingest

3. **异常列表**
   - 最近失败事件
   - 对应模块
   - 失败原因摘要
   - 跳转建议

4. **Langfuse 跳转区**
   - 打开 Langfuse
   - 说明哪些细节在外部观测系统查看

#### 设计原则

- 不在 FinModPro 内复刻 Langfuse 全量 UI
- 只做“中台级摘要 + 跳转”
- 避免页面看起来像空壳监控页

### 6.4 `/admin/llm/knowledge` 知识库接入页

#### 页面目标

让管理员理解知识库入口不是“有没有上传文件”，而是“解析链路有没有正确工作”。

#### 页面模块

1. **解析能力状态**
   - TXT 本地解析
   - PDF Unstructured 解析
   - DOCX Unstructured 解析
   - PDF fallback 状态

2. **近期解析任务摘要**
   - 成功数量
   - 失败数量
   - 最近失败原因

3. **文档类型说明**
   - 每种类型当前走哪条解析链路
   - 哪些类型支持 fallback
   - 哪些类型失败后直接中止

4. **入库链路提示**
   - 解析 -> chunk -> embed -> index 的阶段说明

#### 设计重点

- 让管理员知道 Unstructured 确实已接入
- 把“PDF / DOCX 不是同一条失败语义”展示清楚

### 6.5 `/admin/llm/fine-tunes` LLaMA-Factory 页

#### 页面目标

把当前微调能力从“登记表单”升级为“任务流水线控制面”。

#### 页面模块

1. **任务列表**
   - `pending / running / succeeded / failed`
   - 基础模型
   - 数据集
   - 策略
   - 最近更新时间

2. **任务详情抽屉 / 详情面板**
   - 训练参数
   - 数据集信息
   - runner 名称
   - 时间线
   - 指标摘要

3. **导出包与 artifact 区**
   - export path
   - artifact path
   - artifact manifest

4. **回调与运行状态区**
   - callback 状态
   - 最近 heartbeat
   - external job id
   - failure reason

5. **回流结果区**
   - deployment endpoint
   - deployment model name
   - 已生成的 LiteLLM alias
   - 是否已注册为候选模型

#### 页面定位

- 不是完整训练编排器
- 是当前阶段控制面
- 页面要明确表达：真实训练执行仍依赖外部 runner / 基础设施

## 7. 关键数据映射

### 7.1 总览页数据来源

- 模型配置：`ModelConfig`
- 当前启用模型：运行时 active 配置
- 微调任务：`FineTuneRun`
- 观测摘要：现有 trace / service 侧聚合接口
- 解析状态：知识库 ingest / parser 侧摘要接口

### 7.2 LLaMA-Factory 页数据来源

- 任务主表：`FineTuneRun`
- 导出包详情：现有 export detail 接口
- runner spec：现有 runner-spec 接口
- 回调结果：callback 回写字段
- LiteLLM alias 结果：artifact manifest 与候选模型注册结果

### 7.3 非目标

本设计不要求页面直接读取外部 Langfuse 全量原始 trace，也不要求直接接管 LiteLLM 原生管理面板。

## 8. 交互与状态设计

### 8.1 加载态

- 总览卡使用骨架屏或保留占位，避免内容跳动
- 表格页使用行级 loading，而不是整页白屏
- 所有刷新按钮在请求期间禁用

### 8.2 空态

- 无模型配置：提示先创建并启用模型
- 无微调任务：提示当前尚未创建 LLaMA-Factory 任务
- 无观测数据：提示尚未出现调用或 Langfuse 尚未配置

### 8.3 错误态

错误必须尽可能靠近业务对象表达：

- LiteLLM 不可达：展示在基础设施状态卡和模型相关操作区
- Langfuse 未配置：展示为“观测受限”，而不是整页失败
- Unstructured 失败：展示最近失败类型与失败原因摘要
- runner 心跳超时：展示在 LLaMA-Factory 任务列表和详情页

### 8.4 权限态

- 整个 LLM 中台仅管理员可见
- 非管理员不显示导航项，也不允许路由直达
- 页面内部不做额外角色分裂，避免同页多版本复杂度

## 9. 视觉与 UX 规则

### 9.1 风格

- 偏金融、克制、专业
- 以蓝色系为主，谨慎使用绿色作为成功辅助色
- 不使用 emoji 作为图标
- 不使用装饰性渐变 hero 或伪趋势图

### 9.2 组件形态

- 以状态卡、摘要卡、表格、详情抽屉、时间线为主
- 避免一个页面同时出现大量不同视觉语法
- 图标使用统一 SVG 图标集

### 9.3 可访问性

- 保证文字对比度
- 所有交互元素提供清晰 focus 态
- 图标按钮需有文本或 aria label
- 页面在 1024px 以上信息密集，在窄屏下改为纵向堆叠

## 10. 实施策略

### 10.1 第一阶段

- 先完成路由、导航和总览页
- 拆分现有模型配置页到新路由体系
- 新增 LLaMA-Factory 独立页

### 10.2 第二阶段

- 新增观测页与知识库接入页
- 在总览页接入真实摘要数据

### 10.3 第三阶段

- 评估是否将评测结果进一步与 LLM 中台联动
- 评估是否需要更细粒度的微调详情页

## 11. 验证要求

- 管理员能从侧边栏稳定进入 LLM 中台
- 非管理员看不到 LLM 中台入口且无法访问相关路由
- 总览页能清晰表达四类基础设施的接入与状态
- 模型配置页、知识库接入页、观测页、LLaMA-Factory 页职责边界明确
- LLaMA-Factory 页能完整表达任务状态、回调状态和 alias 回流状态
- 空态、失败态、加载态在关键页面都有明确表达

## 12. 明确结论

本次不应继续把能力堆进现有“模型配置”页，而应将其升级为 **管理后台中的 LLM 中台体系**。

最重要的结构决策是：

1. 先做一个能让人“感知到接入”的总览页
2. 将模型、观测、知识库接入、LLaMA-Factory 拆成独立子页
3. 将 LLaMA-Factory 明确设计为独立页面，而不是微调表单的附属区块
4. 保持评测结果暂时独立，但与中台形成跳转联动
