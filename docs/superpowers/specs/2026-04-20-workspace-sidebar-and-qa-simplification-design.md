# FinModPro 业务工作区侧边栏与问答页简化设计

## 目标
本次设计只处理业务工作区中最影响可用性的两类界面问题：

- 业务工作区侧边栏当前被压成仅图标轨，用户无法直接判断各入口含义。
- 智能问答页顶部堆叠了过多说明性信息，主对话区没有真正成为“连续工作面”，视觉上存在内容上缩、首屏浪费的问题。

目标是把业务工作区收束成更清晰的“左侧导航 + 单一问答工作面”结构，让用户进入页面后优先看到可识别的导航和可持续对话区域，而不是重复标签与状态说明。

## 现状问题
结合当前前端结构，问题已经比较明确：

- [`frontend/src/style.css`](/root/finmodpro/frontend/src/style.css) 为业务工作区单独设置了 `--workspace-sidebar-width: 92px`，并在工作区场景下隐藏了品牌文案、分组标题和导航文字，使侧边栏退化成仅图标形式。
- [`frontend/src/components/ui/AppSidebar.vue`](/root/finmodpro/frontend/src/components/ui/AppSidebar.vue) 实际仍保留完整的分组与文案结构，只是被工作区样式全部隐藏，说明问题主要来自壳层样式策略，而不是导航信息缺失。
- [`frontend/src/components/FinancialQA.vue`](/root/finmodpro/frontend/src/components/FinancialQA.vue) 在问答页顶部同时展示“智能问答”、会话状态、数据集范围、会话元信息、摘要等多层文案，造成首屏认知负担偏重。
- 问答页主体虽然已经使用抽屉承载历史会话和记忆，但聊天卡片高度与内部布局仍偏保守，消息区域没有吃满剩余空间，看起来像被压缩在页面上半部。

## 设计结论
本次采用“展开型业务工作区 + 简化型问答头部”的方案。

页面优先级调整为：

1. 左侧清晰可读的业务导航
2. 问答页顶部轻量操作条
3. 满高聊天消息区
4. 稳定固定的输入区

## 方案对比
### 方案 A：展开型左侧导航
这是本次采用的方案。

优点：

- 直接解决“只看见图标、不知道入口含义”的核心问题。
- 与管理台壳层认知保持一致，减少工作区内部的导航学习成本。
- 改动主要集中在现有样式和问答页结构，不需要引入新的折叠状态机。

代价：

- 会占用比当前窄栏更多的横向空间。

之所以仍然选择该方案，是因为当前用户痛点首先是“不可识别”，不是“空间不足”。

### 方案 B：默认展开，可手动折叠
优点：

- 同时兼顾可读性与空间效率。

问题：

- 会引入额外的折叠状态、按钮、记忆逻辑和响应式细节。
- 对当前任务来说属于过度设计。

本次不采用。

### 方案 C：维持窄栏，在内容区补文字导航
优点：

- 壳层改动最小。

问题：

- 信息被拆成两处，导航认知不集中。
- 不能真正解决左侧导航本身不可读的问题。

本次不采用。

## 业务工作区侧边栏设计
### 导航结构
业务工作区继续复用现有 `AppSidebar` 组件与分组逻辑，不重做信息架构：

- `核心工作`
- `工作记录`

保持现有导航项顺序与路由关系不变，只恢复业务工作区场景下被隐藏的可读文案。

### 壳层宽度
[`frontend/src/style.css`](/root/finmodpro/frontend/src/style.css) 中业务工作区专用宽度不再使用极窄图标轨策略，而是回到与常规导航一致的可读宽度区间。

调整方向：

- 让品牌文案恢复显示。
- 让分组标题恢复显示。
- 让每个导航项始终显示图标和文字。
- 保持当前桌面端纵向导航、移动端横向滚动的响应式结构，不新增折叠按钮。

### 视觉方向
延续仓库当前的克制、机构化工作台风格，不新增装饰性 hero 或夸张视觉效果：

- 继续使用现有浅色表面、细边框和低对比阴影体系。
- 让工作区侧栏更接近“稳定工具栏”，而不是实验性图标轨。
- 当前激活项仍保留清晰高亮，但避免过重位移和视觉噪音。

## 问答页结构设计
### 顶部操作条
问答页顶部只保留真正需要频繁操作的入口：

- `历史会话`
- `记忆`
- `新对话`

以下头部信息移除或弱化，不再占据主视觉：

- “智能问答”眉标
- 新对话状态标签
- “数据集范围”标题文案
- “全部数据集”这类会话元标签
- “等待首轮问答”这类解释性状态文案
- 当前会话摘要

顶部不再承担“介绍页”角色，而只承担工作入口角色。

### 数据集筛选
数据集筛选能力保留，但从头部主标题区下沉，避免与主操作竞争注意力。

建议位置：

- 输入区上方的弱化工具位，或
- 输入区左侧的次级控件位

设计原则：

- 它仍然是可用功能，但不应成为问答页首屏的主信息。
- 新会话和已有会话都继续兼容当前 `dataset` query 与 `contextFilters` 逻辑。

### 聊天主画布
问答页改成更纯粹的三段式：

1. 顶部操作条
2. 满高消息区
3. 底部固定输入区

实现目标：

- 让消息区持续占据剩余高度，不再看起来像缩在卡片上半部。
- 输入区继续固定在底部，形成稳定的操作底座。
- 消息宽度继续足够舒展，但不引入额外常驻侧栏。
- 历史会话抽屉与记忆抽屉能力保持不变，只保留入口按钮。

### 首屏空态
当前默认系统欢迎语不再以正式消息流的强存在感长期占位。

建议调整为更弱的空态提示：

- 仅在没有真实对话消息时展示
- 样式上弱于正式消息气泡
- 不影响开始首轮提问后的消息流阅读节奏

## 状态与数据流
本次不改变现有前端状态边界与后端接口契约。

保留如下行为：

- 历史会话仍由 `FinancialQA` 作为主状态源。
- `session` 路由 query 仍用于打开指定会话。
- `dataset` 路由 query 与 `contextFilters` 仍保持兼容。
- 历史会话和记忆抽屉仍由当前按钮触发。

变化仅在界面层级与展示权重，不新增新的业务状态机。

## 异常处理
本次仍沿用现有错误处理策略，不新增静默回退：

- 数据集加载失败时，继续按当前逻辑保留已有提示或降级行为。
- 历史会话加载失败时，继续展示当前可读错误。
- 记忆抽屉加载失败时，维持现有错误反馈。

界面简化不能以隐藏失败状态为代价。

## 测试策略
### 前端
需要补充或更新前端测试，覆盖：

- 业务工作区侧边栏在桌面端始终显示导航文字。
- 问答页头部不再渲染被明确移除的冗余文案。
- 历史会话、记忆、新对话三个操作入口仍可正常触发。
- 新会话与历史会话切换时，数据集筛选逻辑仍保持兼容。

### 构建验证
- `cd frontend && npm test`
- `cd frontend && npm run build`

## 验收标准
- 业务工作区左侧导航在桌面端始终可读，不再只有图标。
- 问答页顶部只保留历史会话、记忆、新对话三类主操作入口。
- 问答页首屏不再出现“智能问答 / 数据集范围 / 全部数据集 / 等待首轮问答”等被点名的冗余头部文案。
- 聊天主区域在桌面端视觉上明显铺满，消息区不再出现异常上缩感。
- 历史会话抽屉、记忆抽屉和数据集筛选功能仍然可用。

## 非目标
- 不重做业务工作区整体导航信息架构。
- 不调整知识库、风险、舆情等其它业务页面的内容结构。
- 不引入侧边栏折叠能力或新的个性化布局配置。
- 不改动后端 chat / memory / dataset 接口协议。
