# FinModPro 平台收尾方向与开发重点

更新日期：2026-04-13

本文档用于明确 `finmodpro` 后续收尾阶段的总体方向、功能边界和开发优先级。目标不是把所有设想功能同时铺开，而是在现有基础上继续完善平台，形成结构完整、主线清晰、可持续扩展的金融风控大模型应用平台。

## 1. 收尾阶段总方向

收尾阶段采用以下总体策略：

- 主策略：`主线深化型`
- 辅策略：`少量场景扩展`
- 不采用：纯展示型堆页面、纯运维型治理优先、一次性铺开过多新模块

即：

- 继续围绕当前已经成型的能力做深
  - 知识库
  - RAG 检索增强问答
  - 风险事件抽取与报告
  - 模型配置与评测
  - 用户/权限/后台管理
- 在平台主线补强之后，允许新增 `1` 个高辨识度业务场景模块
  - 优先推荐：`舆情分析`

## 2. 当前平台定位

`finmodpro` 当前已经具备以下平台雏形：

- 用户认证与 RBAC
- 知识库文档上传、入库、状态跟踪
- 基于知识库的智能问答
- 多轮问答与会话记录
- 风险信息抽取、审核、报告生成
- 管理台仪表盘、用户管理、模型配置、基础评测

这说明项目已经从“演示原型”进入“可继续平台化完善”的阶段。后续开发重点不应再是零散补页面，而应转为：

- 补平台骨架
- 补数据与模型链路
- 补可验证的效果闭环
- 谨慎扩展新场景

## 3. 收尾阶段的核心目标

后续开发应围绕以下四个核心目标推进。

### 3.1 完善知识增强主线

把当前“文档库 + 入库 + 问答”升级为更清晰的平台能力：

- 数据集管理
- 文档归档与检索范围控制
- 文档版本与来源追踪
- 知识分类、标签、筛选
- 更明确的入库链路状态与失败原因

目标是让知识库不只是“能上传”，而是具备真正的平台管理能力。

### 3.2 完善领域适配主线

项目题目与开题说明中明确提到了：

- 金融领域适配
- LoRA 等参数高效微调
- 基于金融数据增强模型能力

因此后续必须补齐“领域模型适配”这一条主线，至少形成最小闭环：

- 微调数据准备
- 微调任务记录
- 微调模型注册
- 微调模型切换或启用
- 微调前后效果对比

如果这条线一直缺失，平台会更像“通用模型 + RAG 应用”，而不是“金融领域大模型应用平台”。

### 3.3 完善风控应用闭环

当前最成熟的业务能力是：

- 智能问答
- 风险事件抽取
- 风险审核
- 风险报告

后续应继续做深这条链路，而不是同时跳到过多高复杂场景。重点包括：

- 风险事件筛选与统计
- 报告导出
- 会话导出
- 风险图表展示
- 更完整的审核和结论沉淀

### 3.4 控制扩展节奏

平台允许新增场景，但必须满足两个前提：

1. 不打断现有主线补强
2. 新场景能够复用现有底层能力

因此新增场景优先选：

- 舆情分析

暂不建议优先投入：

- 反洗钱
- 欺诈检测
- 复杂交易网络分析
- 重型实时流式接入体系

这些方向数据依赖重、业务复杂度高、实现成本大，容易稀释项目主线。

## 4. 推荐的开发路线

推荐采用：

`主线深化型 + 少量场景扩展`

具体含义如下：

### 4.1 先做主线深化

优先补齐以下平台骨架：

- 数据集管理
- 版本溯源
- 检索控制
- 模型评测
- 微调/领域适配链路
- 风险闭环增强

### 4.2 再做场景扩展

在主线能力稳定后，新增 `1` 个更容易展示平台价值的业务模块。

优先选择：

- 舆情分析

原因：

- 与现有文本处理、知识库、风险抽取链路天然相近
- 数据形态一致，开发成本低于合同审查、AML、欺诈检测
- 展示效果好，适合作为平台能力扩展案例

### 4.3 不建议的路线

以下路线不建议作为收尾主方向：

- 同时新增多个新场景模块
- 先做大规模监控审计治理，再回头补主线
- 为了“看起来功能多”而继续扩展空壳页面
- 一次性上完整企业级数据接入能力

## 5. 后续开发重点模块

后续开发分为 `P0 / P1 / P2` 三个层级。

## 5.1 P0：必须优先完成的主线模块

这些能力决定平台骨架是否完整，应最优先推进。

### P0-1 数据集管理

目标：

- 让知识库从“文档列表”升级为“数据集 + 文档”的双层结构

建议补充能力：

- 创建数据集
- 选择当前数据集
- 文档归属到指定数据集
- 问答与检索按数据集生效
- 风险抽取按数据集或文档范围执行

原因：

- 这是开题报告里“创建/选择数据集”的直接落地点
- 也是后续版本管理、评测、微调数据组织的基础

### P0-2 文档版本管理与溯源

目标：

- 明确“知识从哪里来、经过哪些处理、当前使用哪个版本”

建议补充能力：

- 文档版本号
- 文档替换/新版本上传
- 来源信息记录
- 处理步骤记录
- 当前激活版本标识

可先不做复杂图谱式血缘分析，但至少要先有结构化溯源信息。

### P0-3 RAG 能力增强

目标：

- 让问答和检索更加“平台化、可控化”

建议补充能力：

- 按数据集、标签、时间范围检索
- 检索参数可配置
- 更清晰的引用展示
- 检索命中与未命中解释
- 问答与知识来源关联记录

### P0-4 风险闭环增强

目标：

- 把“抽取 -> 审核 -> 报告”从已有链路做成完整业务闭环

建议补充能力：

- 风险事件筛选和排序
- 按风险类型/等级统计
- 风险图表可视化
- 报告导出
- 报告下载或分享

### P0-5 模型评测体系增强

目标：

- 让平台具备可量化验证模型能力的基础设施

建议补充能力：

- 问答评测指标
- 风险抽取评测指标
- 模型版本维度对比
- 基础模型 / RAG / 微调模型对比
- 更清晰的评测记录页

### P0-6 LoRA / 微调最小闭环

目标：

- 补齐项目题目中的“领域模型适配”主线

建议补充能力：

- 微调数据集管理
- 微调任务记录
- 微调产物登记
- 模型版本注册
- 微调模型启用与回滚
- 微调前后评测对比

说明：

- 即使训练流程不能完全在线化，也应至少形成平台内可管理的最小闭环
- 可以采用“训练在外部脚本执行，结果回写平台登记”的过渡方案

## 5.2 P1：强烈建议补齐的能力

这些能力会显著提升平台完整度，但优先级低于 P0。

### P1-1 历史会话能力补全

当前问题：

- 独立历史页闭环还不完整

建议补充：

- 历史会话列表页真实数据加载
- 会话详情回看
- 会话搜索/筛选
- 会话导出

### P1-2 任务监控与失败重试

建议补充：

- 入库任务状态页
- 失败任务重试
- 抽取任务执行记录
- 关键后台任务可见性

### P1-3 操作日志与审计

建议补充：

- 用户关键操作记录
- 管理员操作记录
- 文档处理与模型切换日志
- 简单导出能力

### P1-4 报告与结果导出

建议补充：

- 风险报告导出 PDF/Markdown
- 风险事件明细导出
- 问答会话导出

## 5.3 P2：允许扩展的新增场景

这些能力在主线稳定后再推进。

### P2-1 舆情分析模块

优先级最高的新场景。

建议能力：

- 舆情类文本导入
- 情感倾向/风险倾向识别
- 舆情风险聚合
- 时间趋势图
- 关联风险事件生成

原因：

- 能复用当前文本入库、问答、风险抽取基础设施
- 与金融风控主题贴合
- 相比合同审查、AML、欺诈检测更容易落地

### P2-2 合同审查模块

可作为第二选择，但优先级低于舆情分析。

建议原因：

- 可复用文档上传与文本理解链路
- 对大模型能力展示直观

不建议与舆情分析同时启动，避免分散开发资源。

## 6. 当前阶段不建议优先投入的方向

以下能力不是不能做，而是不适合作为当前收尾阶段的优先事项：

- 全量多源异构实时数据接入
- 消息队列级实时流处理体系
- 完整企业级资源告警平台
- 自动化模型漂移检测全套
- AML 深度行为网络分析
- 欺诈检测复杂规则系统
- 过重的合规自动生成体系

原因：

- 对数据和业务依赖过重
- 工程投入明显高于当前平台阶段
- 容易让项目从“平台完善”变成“范围失控”

## 7. 推荐的开发阶段划分

为了避免后续开发继续发散，建议按四个阶段推进。

### 阶段 A：平台骨架补齐

优先完成：

- 数据集管理
- 文档版本管理
- 溯源信息记录
- 历史会话补全

交付结果：

- 平台数据组织方式清晰
- 知识资产管理能力成型

### 阶段 B：知识与风险主线增强

优先完成：

- RAG 检索控制增强
- 风险事件筛选统计
- 风险图表
- 报告导出
- 任务监控与失败重试

交付结果：

- 问答与风险闭环更完整
- 平台不再只是“能跑通”，而是“可管理、可复盘”

### 阶段 C：领域适配补齐

优先完成：

- 微调数据准备
- LoRA/微调任务记录
- 模型版本管理增强
- 微调评测对比

交付结果：

- 题目中的“金融领域适配”有真实落点
- 平台从“RAG 应用”升级为“领域模型应用平台”

### 阶段 D：新增场景扩展

优先完成：

- 舆情分析模块

交付结果：

- 平台具备至少两个较完整的金融风控应用场景
  - 智能问答
  - 风险抽取/报告
  - 舆情分析

## 8. 后续开发原则

收尾阶段需要遵循以下原则：

- 优先补主线，不优先堆展示页
- 所有新增能力尽量复用已有知识库、RAG、模型配置、评测体系
- 每新增一个场景，都要能落到可验证的数据链路和效果验证
- 避免同时开启多个高复杂模块
- 让平台结构越来越清晰，而不是越来越分散

## 9. 结论

`finmodpro` 后续收尾阶段的推荐方向已经明确：

- 以 `主线深化` 为主
- 以 `少量场景扩展` 为辅
- 优先补齐平台骨架、数据组织、版本溯源、RAG 控制、风险闭环、模型评测与 LoRA/微调链路
- 在主线稳定后，优先新增 `舆情分析` 模块作为扩展场景

这条路线能够同时满足三个目标：

- 与当前代码基础连续
- 与项目题目和论文主线一致
- 能持续把项目从“可运行原型”推进到“结构完整的平台”
